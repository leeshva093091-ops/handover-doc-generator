# PRD — 프로젝트 인수인계 문서 자동 생성기 (가칭)

## 1. 프로젝트 개요
- **해결하려는 문제**: 인수인계가 사람·시점마다 들쭉날쭉해 신뢰할 수 없음. 문서가 없거나, 있어도 양식 제각각, 금방 낡고, 핵심 주의사항은 개인 머릿속에만 있어 새 담당자가 매번 처음부터 헤맴.
- **배경**: 매번 반복 발생하는 문제라, 한 번 자동화하면 계속 이득.
- **기대 변화**: 며칠씩 코드 뒤지고 전임자에게 묻던 것 → 문서 한 장으로 30분 내 설치·실행 도달. 모든 프로젝트가 같은 양식을 가져 소통·이해가 쉬워짐.

## 2. 사용자 분석
- **주 사용자**: 인수인계를 넘기는 사람(전임)과 받는 사람(후임) 양방향. "뽑기 쉽게 + 읽기 쉽게" 둘 다 중요.
- **사용 맥락**: 담당자 교체/퇴사 시점, 배포·프로젝트 마무리 시 갱신. 한 번 쓰고 끝이 아니라 주기적으로 다시 돌리는 패턴.
- **현재 해결 방식과 한계**: 코드·주석·전임자 구두 설명에 의존 → 귀찮아서 누락, 어쩌다 만든 문서도 양식 제각각이라 한눈에 안 들어옴.

## 3. 기능 요구사항
**핵심 기능 (우선순위 순)**
1. **자동 분석** — README·의존성·실행 스크립트·디렉터리 구조를 코드에서 직접 읽어냄 (도구의 심장)
2. **정형 양식 출력** — 무엇을 넣든 동일한 인수인계 템플릿에 채워 출력
3. **주의사항 추출** — 환경변수·포트·외부 의존성·폐쇄망 설정 등 함정 잡기
- (지원) 소스 입력(GitHub URL/로컬 폴더), 문서 저장·내보내기

**차별점**: 사람 손 거의 안 대고(누락·귀찮음 제거) + 항상 같은 양식 + 외부 서비스·AI 없이 폐쇄망 로컬 파이썬만으로 동작.

## 4. 비기능 요구사항
- **사용 환경**: 웹 화면 메인(브라우저에서 URL/폴더 입력 → 실행), CLI 서브(자동화·배치). 모두 폐쇄망 내 로컬 동작. (웹 구동 방식 — 표준 라이브러리/Flask/정적 HTML — 추후 결정)
- **데이터 민감도**: 주로 내부 인프라 정보(IP·포트·경로, 내부망이라 허용). 비밀번호·API 키·DB 접속정보가 섞일 수 있음 → 차단보다는 "민감정보 주의" 표시로 잡아줌.
- **제약**: 혼자 짬 내서 개발(시간 빠듯), 폐쇄망이라 추가 패키지 최소화(표준 라이브러리 위주).

## 5. 성공 지표
- **성공 모습**: 새 담당자가 문서 한 장으로 30분 내 설치·실행 도달, 나아가 인수인계 문서를 손으로 안 쓰게 됨.
- **핵심 지표**: 온보딩 시간 (설치·실행까지 걸린 시간, 목표 30분 이내).

## A. 사용자 여정 (User Journey)
1. 전임자가 웹 화면을 열고 프로젝트 폴더 경로(또는 GitHub URL)를 입력
2. "분석" 버튼 클릭 → 도구가 코드·설정·문서를 스캔
3. 결과 미리보기: 설치/실행/주의사항이 정형 양식으로 채워져 표시
4. 민감정보(키·접속정보) 감지 시 "주의" 배지로 강조 → 전임자가 확인·보정
5. 문서를 Markdown/HTML로 저장·공유 → 후임자가 그 문서만 보고 30분 내 실행

## B. 기능별 상세 사양 (입력 → 동작 → 출력)
### 1. 자동 분석
- **입력**: 로컬 폴더 경로 또는 GitHub URL
- **동작**: README, 의존성 파일(requirements.txt·package.json·pom.xml 등), 실행 스크립트(run.sh·Makefile·Dockerfile), 디렉터리 구조 파싱
- **출력**: 구조화된 메타데이터(언어, 의존성 목록, 실행 진입점, 포트 등)

### 2. 정형 양식 출력
- **입력**: 1번의 메타데이터
- **동작**: 고정 템플릿 슬롯(개요/설치/실행/환경/주의사항)에 매핑
- **출력**: 항상 동일한 섹션 구조의 인수인계 문서

### 3. 주의사항 추출
- **입력**: 코드·설정 텍스트
- **동작**: 환경변수·포트·외부 의존성·하드코딩된 키/접속정보 패턴 탐지
- **출력**: "주의" 목록 + 민감정보 위치 표시(값은 마스킹 권장)

(지원) 소스 입력 / 문서 저장·내보내기(Markdown·HTML)

## C. 단계별 개발 우선순위
**[MVP — 가장 먼저]**
- CLI로 로컬 폴더 분석 → 정형 Markdown 문서 출력
- 자동 분석(의존성·실행 스크립트·README) + 정형 양식
- 파이썬 표준 라이브러리만으로 구현

**[다음 단계]**
- 주의사항 추출(환경변수·포트·민감정보 패턴) + 마스킹
- 웹 화면(입력 폼 + 결과 미리보기) — 구동 방식 이때 확정
- GitHub URL 입력 지원

**[추후]**
- 다국어/다언어 프로젝트 폭넓게 지원
- 같은 프로젝트 재분석 시 문서 갱신(diff)
- HTML 외 내보내기 포맷 확장

## D. 위험 요소
1. **분석 정확도**: 프로젝트마다 구조·실행법이 제각각 → 자동 추출이 빗나갈 수 있음. 대응: 사람이 미리보기에서 보정 가능하게.
2. **민감정보 오탐/누락**: 키 패턴 탐지가 완벽하지 않음 → 노출 위험. 대응: 보수적으로 "의심 항목"까지 넓게 표시 + 마스킹 기본값.
3. **폐쇄망 의존성**: 웹 프레임워크 도입 시 .whl 사전 반입 필요. 대응: MVP는 표준 라이브러리만, 웹은 다음 단계에서 결정.
